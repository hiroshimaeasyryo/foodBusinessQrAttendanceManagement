# プロジェクトコンセプト
- このプロジェクトは`https://github.com/hiroshimaeasyryo/internQrAccessControl.git` のフォークである。

# 相違点
- フォーク元プロジェクトとの相違点は以下の通り。
## 目的
- 元プロジェクトはスタッフの入退室管理が目的であったのに対し、本プロジェクトは出勤管理を目的とする。
- 元プロジェクトは入退室におけるスキャン時の正確なタイムスタンプを記録するのに対して、本プロジェクトでは定時を記録する。また、出勤時に退勤時刻も一緒に記録し、イレギュラーがある場合だけ手入力を行う仕様とする。
## 記録方法
- 本プロジェクトでは、スタッフの属性(property)によって記録する時刻が異なる。
    - 属性が「社員」のスタッフは、「11:00-24:00(11h勤務, 2h休憩)」(デフォルト)か「18:00-24:00(6h勤務, 休憩なし)」のいずれかを選択する。その他イレギュラー対応として、任意の時間(出勤時刻、退勤時刻、休憩時間)を入力できるフィールドを用意しておく。ただし退勤時間は日を跨ぐことも想定され、その場合にはわかりやすさのため「01:00」ではなく「25:00」などと24時+α表記とする。
    - 属性が「契約社員」の場合は「18:00-22:00(4h勤務, 休憩なし)」(デフォルト)を記録し、イレギュラー対応として任意の時間(出勤時刻、退勤時刻、休憩時間)を入力できるフィールドを用意しておく。同様に24時+α表記とする。
    - 属性が「業務委託」の場合は何も記録しない。
## 認証およびユーザー動線
- フォーク元プロジェクトは、スタッフの選択=>生年月日(8桁数字)の入力による認証=>入室/退室の選択=>QRコードのスキャンというユーザー動線であった。
- 本プロジェクトでは、誕生日(4桁数字)の入力により現れる候補スタッフの選択=>QRコードのスキャンという動線とする。
    - この時、誕生日の入力後に候補スタッフを探すと時間がかかるため、初めから誕生日とスタッフの組み合わせをフロントエンドで用意しておき(キャッシュまたは非同期通信等でページを読み込んだ時点であらかじめ情報を持っておく)、用意しておいた4桁のパターンと合致次第すぐにスタッフを表示するようにする。
    - このアプリケーションで使用するユーザーはスタッフは数名しかいないため、大量データを想定する必要はない。
## Spreadsheetの構成
- フォーク元のSpreadsheetは以下の構成だった。
    - 「スタッフDB」シート
        - `uuid` カラム
        - `name` カラム
        - `birthdate` カラム
        - `img` カラム
    - 「打刻記録」シート
        - `タイムスタンプ` カラム
        - `スタッフID` カラム
        - `学生氏名` カラム
        - `拠点` カラム
        - `打刻種類` カラム
        - `ユーザーエージェント` カラム
- 本プロジェクトのSpreadsheet(ID: `1BzM547ikvZIjXLEQBktuRxeU0UaiPH7Td1m80tBoSGE`)は次の構成とする。
    - 「スタッフDB」シート
        - `uuid` カラム
        - `name` カラム
        - `birthdate` カラム
        - `property`カラム
    - 「打刻記録」シート（7列の生ログ。列はカラム名で参照し、列の移動・挿入に耐える設計。区分・勤務時間・備考は転記用で賄う）
        - `タイムスタンプ` カラム
        - `スタッフID` カラム
        - `氏名` カラム
        - `日付` カラム（出勤日を格納）
        - `出勤時刻` カラム
        - `退勤時刻` カラム
        - `休憩時間(時間)` カラム（記録値は時間。入力UIは分のまま 90→1.5）
    - 「{STAFF_FAMILY_NAME}」シート(1-2行目がヘッダー行)
        - `日付` カラム
        - `始業` カラム
        - `終業` カラム
        - `休憩時間(時間)` カラム（同上）
        - `休み` カラム(手動記録のため本プロジェクトのスコープ外)
        - `備考` カラム(手動記録のため本プロジェクトのスコープ外)
    - 本プロジェクトでは、「打刻記録」シートと各スタッフの「{STAFF_FAMILY_NAME}」シートに記録を行う。`氏名` カラムには「{STAFF_FAMILY_NAME} {STAFF_LAST_NAME}」のように指名の間に半角スペースが設けられている。
- **スタッフシートの手動修正の反映**: ユーザーがスタッフ別シートの「始業」「就業」「休憩時間(時間)」を手動で修正することがある。その内容を打刻記録に反映するため、`syncStaffSheetToTimestamp(options)` を用意している。実行すると、各スタッフシートの該当列を読み、同じ (日付, スタッフID) の打刻記録行の「出勤時刻」「退勤時刻」「休憩時間(時間)」を上書きする。転記用シートは打刻記録を参照するため、同期後に日次転記（または `runDailyTransfer`）を実行すると転記用にも反映される。オプション: `staffFamilyName`（姓のみ対象）, `startDateStr`, `endDateStr`（日付範囲）。
    - 本プロジェクトではスタッフのアイコン写真は用いない。
    - 別Spreadsheetへの転記用フォーマット（コード内 `TRANSFER_SHEET_COLUMNS`）: `年月`, `スタッフID`, `氏名`, `日付`, `区分`, `出勤開始`, `出勤終了`, `休憩時間(時間)`, `勤務時間(時間)`, `備考`。確定フラグは転記先の別Spreadsheetで使うため、転記用シートには設けない（設ける場合も空欄のまま渡す想定）。同一ブック内に「転記用」シートを1枚設ける。
## 転記用シートの構成と記録方法
- **シート名**: `転記用`（コード内定数 `TRANSFER_SHEET_NAME`）
- **役割**: 別Spreadsheetと同じ列構成でデータを保持し、確認後にコピーするための中間シート。確定フラグは転記先で使用するため本シートには設けない。
- **カラム構成**（1行目ヘッダー。列はカラム名で参照し、列の移動・挿入に耐える）:
  - `年月` … 日付から算出（YYYYMM）。例: 2025-03-10 → 202503
  - `スタッフID` … 打刻記録またはスタッフDBの uuid
  - `氏名` … スタッフ名（姓 名）
  - `日付` … 出勤日（yyyy-MM-dd）
  - `区分` … 「出勤」またはスタッフ別シートの「休み」列の値（有給・欠勤等）
  - `出勤開始` … 出勤時のみ。打刻記録の「出勤時刻」に相当
  - `出勤終了` … 出勤時のみ。打刻記録の「退勤時刻」に相当
  - `休憩時間(時間)` … 出勤時のみ。休みの行は空または 0
  - `勤務時間(時間)` … 出勤時のみ。休みの行は空または 0
  - `備考` … 空または手動入力
- **記録方法**:
  1. **出勤分**: 日次転記バッチが「打刻記録」を走査し、まだ転記用に存在しない (日付, スタッフID) の行を転記用フォーマットに変換して追記。出勤開始＝出勤時刻、出勤終了＝退勤時刻。
  2. **休み分**: 各スタッフ別シート（{STAFF_FAMILY_NAME}）の「日付」列と「休み」列を走査し、休みに値がある行について、(日付, スタッフID) が打刻記録にも転記用にもまだ無い場合のみ、区分＝休みの値・出勤開始/終了・休憩・勤務時間は空で転記用に1行追加。氏名はスタッフDBから uuid→name で解決（シート名は姓のみのため）。
  3. **重複排除**: 転記用シートに既に同じ (日付, スタッフID) の行がある場合は追加しない。打刻記録と休みの両方に同じ日・同一スタッフがある場合は「出勤」を優先し1件だけ転記する。
  4. **過去の変更反映**: `runDailyTransfer(対象日, { backwardDays: N })` で N 日分さかのぼって、転記用の既存行を「打刻記録」と各スタッフシートの現在の内容と照合する。区分（休み列）の変更・スタッフシートへの行の挿入などがあった場合、該当する転記用の行を上書きする。定時実行では `runDailyTransferScheduled()` を使い、内部で `backwardDays: 60` を指定している。
- **初期化**: 転記用シートが存在しない場合はバッチまたは初回実行時に作成し、1行目に上記カラムをヘッダーとして設定する。
- **実行タイミング**: 毎日定時には GAS の時間駆動トリガーで `runDailyTransferScheduled` を実行する（例: 日タイマー 午前0時〜1時）。手動の場合は `runDailyTransfer()`（対象日省略時は昨日）または `runDailyTransfer('yyyy-MM-dd', { backwardDays: 60 })` で対象日と過去反映日数を指定可能。`backwardDays > 0` で実行する場合、転記の前に同じ日付範囲で「スタッフシート → 打刻記録」の同期（`syncStaffSheetToTimestamp`）が自動で行われる。
## 転記用シートの別スプレッドシートへの同期
- **転記先**: 転記用シートの内容を、別スプレッドシート（ID: `1W2mWjdQGvPal8D06N1s1ompdicgc3mL5Cc6iRnhXiPs`）の**既存シート「勤怠明細」**に同期する。転記先のオーナーは別アカウントでも、編集権限があれば GAS から書き込み可能（コード内定数 `EXTERNAL_TRANSFER_SPREADSHEET_ID`、シート名 `EXTERNAL_ATTENDANCE_SHEET_NAME`＝勤怠明細）。**勤怠明細から転記用への逆流は行わない。**
- **同期方法**: 転記用の各行について、転記先「勤怠明細」に同じ (日付, スタッフID) の行があればその行を更新、なければシートの最終行に追加する。
- **区分の変換**: 勤怠明細へ書き込む前に、次のように区分値を変換する（コード内 `KUBUN_MAP_TO_EXTERNAL`）。それ以外の区分はそのまま書き込む。
  - `年次有給休暇` → `有給休暇`
  - `半休（午前休）` → `午前休`
  - `半休（午後休）` → `午後休`
- **定時バッチでの同期**: `runDailyTransferScheduled()` は、日次転記（昨日分の追記＋過去60日分の変更反映）の直後に `syncTransferSheetToExternalSpreadsheet()` を実行する。そのため「過去情報の修正」は転記用に反映されたうえで、同じ実行で勤怠明細にも反映される（既存行は更新、新規は追加）。
- **手動・追加トリガー**: 転記のみ再実行せずに同期だけ行う場合は `syncTransferSheetToExternalSpreadsheet()` を手動実行するか、時間駆動トリガーでこの関数だけを指定するとよい。

## 打刻記録シートと転記用の役割分担（日付・区分・勤務時間・備考について）
転記用シートを設ける場合、打刻記録は「生ログ」に近づけ、転記用で「日付・区分・勤務時間・備考」をまとめて持つ設計にすることができる。
- **日付**: 打刻記録に残すことを推奨する。記録したタイムスタンプは「打刻した日時」であり、出勤日（その日の勤務として扱う日）と異なる場合がある（翌日に打刻したなど）ため、アプリで明示的に出勤日を書き込んでおくと転記バッチがそのまま使えて安全。
- **区分**: 打刻記録では常に「出勤」のみなので、列を設けず転記バッチで転記用に書く際に「出勤」を付与する形にしてもよい。ただし打刻記録にも区分列があると、将来「打刻時に区分を選ぶ」などに拡張しやすい。
- **勤務時間(時間)**: 出勤時刻・退勤時刻・休憩時間から算出できるため、打刻記録には持たず転記用でだけ計算して持つ設計でよい。
- **備考**: アプリからは何も入れない前提なら、打刻記録に備考列は不要。転記用で空欄のまま持てばよい。
- まとめ: **日付は打刻記録に残す**、**区分・勤務時間・備考は転記用だけで賄う**方針を採用。打刻記録の列は「タイムスタンプ, スタッフID, 氏名, 日付, 出勤時刻, 退勤時刻, 休憩時間(時間)」の7列とする。
